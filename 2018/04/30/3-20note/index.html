2018/3/20 note

2018-04-30
  • ngFor directive도 원래는 클래스라서 클래스명은 대문자다
  • 디렉티브 만드는 이유 : 여러개 컴포넌트에서 동일하게 사용되는 뷰에대한 관심사를 가진 기능
  • 툴팁은 대표적인 디렉티브의 대상, 마우스오버하면 설명이 뜬다
  • todolist를 가져온다, 삭제한다같은 기능은 특정 컴포넌트에서만 쓰지 않고 이곳저곳에서 사용할 확률이높다.
  • 서비스는 싱글톤으로 만들어지는 클래스 / 어플전역에서 단하나만 존재하는 인스턴스 / 우리가 만드는게 아니고 프레임워크가 만드는데 이걸 제어권역전이라한다
  • 인스턴스를 컴포넌트가 생성하지 않는다. (생성하면 긴밀한 결합이 된다. 그래서 안함)
  • 컴포넌트가 서비스를 적게 알기 위해 dependency injection을 쓴다.
  • di, ioc같은 패턴이 앵귤라에 녹아있다.
  • 일단은 프레임워크가 지향하는바를 이해하자
  • @Injectable은 뒤에 메타데이터가 없어서 사실 없어도 되지만 붙이는게 약속이다.
  • 서비스 인스턴스를 new로 만들면 생기는 문제
    언제든지 컴포넌트는 바뀐다. 서비스도 바뀐다. 서비스를 대체하자고 했을 때 컴포넌트(서비스를 사용하고자하는 애)도 로직이 바껴야된다. 안티패턴이다.
  • 객체를 생성하는 방법은 여러개,
  • 서비스를 사용하는 컴포넌트는 서비스에 의존한다.
  • 프레임워크는 어플 전체를 아우른다, 프레임워크라서 DI가 가능하다 라이브러리는 기능을 구현하는거뿐,
  • 서비스는 컴포넌트에 등록할수도있고 모듈에도등록이되서 자동적용안해준다.
  • app.module에 사용하는 프로바이더는 어플 전역에서 사용
  • 컴포 프로바이더는 컴포넌트의 자식컴포넌트에서만 유효하다
  • 공유 서비스의 인스턴스는 어플리케이션 내에 단 하나만 있어야한다. (싱글톤)
  • 모듈내에서 프로바이더 설정을 했을때 싱글톤을 지원해준다
  • IoC 제어권역전
    인스턴스는 원래 사용자가 만드는데, 의존관계상의 결합이 긴밀하게 발생한다.
    느슨한 결합을 위해서 프레임워크에게 위임
  • 하나의 인스턴스는 여러개의 타입을 가질 수 있다.
    C는 P 클래스를 상속하면 C는 P,C타입 전부 해당
  • promise, async, awit, generator 다 껄적지근
  • 배열, return : 동기
    ajax통신결과, event: 비동기
    input이 어떤거냐에따라 로직이 크게 바뀐다.
    동기가됐건 비동기가됐건 일관되게 데이터 처리를 하고 싶다.
  • 시간축의 입장에서는 배열, event, ajax 다 순서를 가진 데이타 스트림이다.(배열과 유사한 애들이다)
    input => ===== => output
    모든 input데이타는 데이타스트림이다.
    맵, 필터, 리듀스 써서 처리할 수 있다.(고차함수 비슷하게 체이닝방식에 의해서 코딩할 수 있다.)
  • reactivex
    rxjava, rxswift, rxjs,
    모든 인풋 데이터를 배열처럼 쓰고싶은거다
    옵저버블, 옵저버, 오퍼레이터, 서브젝트, 비헤이비어서브젝트
    옵저버블 : 관측할수있는!
    모든 인풋데이타를 옵저버블로 처리한다.
    그리고 그걸관찰하는 옵저버가있다
    배열도 변한다 그걸 감시하는애가 옵저버
    새로운 클릭이벤트가 추가된다
    리스폰스도 추가된다.
    이벤트처리와 비슷하다.
    옵저버 패턴을 조금 고도화시킨게 리액티브패턴
    Observable.from(this.myArray)
    from이 오퍼레이터
    여러옵저버들이 하나의 옵저버블을 관측할수있다.
