1-29note

2018-02-17
  • 일을 쪼개는 것이 decomposition
  • 프로그래밍 언어는 문법이 있다.
  • 문법에 맞춰서 의미를 명확히 지시하는 것이 중요하다. (semantic)
  • 원천적인 것을 만드는 회사(구글, 삼성)에 가려면 알고리즘을 배워야한다.
  • 자료구조는 데이터를 어떻게 이쁘게 감쌀 것이냐에 대한 것.
  • html, css는 선언형 프로그래밍 언어
    div : 있는 것을 쓴다. mydiv 안됨
  • 프로그래밍은 문법에 부합하는 것은 물론이고 수행하고자 하는 바를 정확히 수행하는 것, 즉 요구사항이 실현(문제가 해결)되어야 의미가 있다.
  • 스크립트언어
    순서가 있고 시간 흐름이 위에서부터 아래로 흐른다.
    한줄 한줄 실행 된다.
    컴파일 과정을 거치지 않고, 자바스크립트 엔진이 코드를 읽어서(파싱) 그 순간순간 바이트코드를 생성해서 브라우저에게 명령한다.
  • 스크립트 엔진은 동시통역사 컴파일러는 번역가
  • 스크립트 언어가 더 느리다.
  • 자바, c는 하이브리드언어
  • c언어는 운영체제가 실행한다. (os한테 직접 명령한다.)
  • 자바스크립트는 브라우저가 실행한다. (os위에서 구동하는 브라우저)
  • 브라우저가 어떻게 동작하는지 알아야 한다.
  • html5의 등장으로 자바스크립트의 사용 범위가 넓어졌다.
  • 플래시는 브라우저에 플러그인을 깔아야 한다.
  • html5는 표준 환경 하에서 기능들이 실행되도록 한 것(플러그인 없이도)
  • 자바스크립트가 플래시가 하던거 다한다.
  • 구글이 v8 엔진을 만들어서 기여를 많이 했다. (자바스크립트 성능 대폭 향상)
  • v8 장착한 크롬으로 ie를 무너뜨렸다.
  • google map : ajax로 어플처럼 동작하는 브라우저를 만들었다. (스크롤 되면 새로운 지도가 이어짐)
  • 자바스크립트는 원래 브라우저에서만 동작했으나, nodejs가 만들어져서 브라우저 밖에서도 쓸 수 있게 되었다.
  • v8이 있었기 때문에 nodejs가 탄생할 수 있었다.
  • 자바스크립트의 본 목적은 간단한 필드체크 등의 수행이었기 때문에 태생적 한계가 있고 타 프로그래밍 언어와 비교했을 때 이상하다.
  • jquery가 각광받은 이유는 DOM 컨트롤이 쉬워서
  • ajax는 자바스크립트에 있지만 jquery ajax가 편리했다.
  • jquery는 직접 DOM을 컨트롤 하기 때문에 html이 변경되면 jquery 로직도 변해야 한다.
  • 모던 어플리케이션에서는 jquery 안쓰고 더 우아한 방법을 쓴다.
  • 자바스크립트는 c, java, awk, perl, python의 영향을 받았다.
  • 인터프리터 언어는 컴파일 하지 않고 엔진이 파싱을 통해서 직접 수행한다. (스크립트 언어와 비슷한 의미다.)
  • 멀티패러다임 언어: 자바스크립트는 명령형 함수형 프로토타입 기반의 객체지향형 언어다.
  • 언어의 자유도가 높아지면 사용하기가 더 까다롭다.
  • 자바는 클래스 기반 객체지향 언어다.
  • 자바스크립트는 프로토타입 기반이며 클래스가 없었다. 근데 es6에 생겼다. 클래스처럼 동작하지만 내부적으로는 프로토타입으로 동작한다. (문법적 설탕)
  • 함수형 : 함수의 매개변수 또는 리턴형으로 객체를 반환하거나 받을 수 있는 언어
  • 자바는 자바스크립트를 따라가고(람다를 쫓아간다 ??) 자바스크립트는 자바를 따라간다.
  • 브라우저는 어떤 os에도 다 있다. 크로스 플랫폼이 된다. os 신경 안써도 된다. ??????
  • 웹페이지인데 앱처럼 동작하는 spa가 대세다.
  • 비영리단체(ecma)에서 자바스크립트 표준을 관리해준다.
  • 자바스크립트의 정식명칭은 ecmascript
  • ecma-262 : 이런식으로 만들자라는 명세서(설명서, 스펙)
  • xmlhttprequest : aka ajax
  • es4는 리젝트됨
  • 익스플로러 전성기는 1999년이었고 발언권이 쎘다.
  • 결국 구글이 브라우저를 쥐락펴락하고 있다.
  • angular는 익스9버전 지원
  • 웹퍼블리싱을 한다면 ie9는 갖고 있어라??
  • babel은 문법을 바꿔준다. (익스8 지원하려고 할 때 es3로 바꿔줌)
  • http, https : 서버와 데이터를 주고받을 때의 통신방법
  • 대부분의 웹서버는 파일명을 명시하지 않으면 index.html을 준다.
  • request : html 파일을 요청한다. css, js도 온다.
  • 서버가 파일을 한줄 한줄 읽어서 문자화된 메모리를 넘긴다.
  • / : root
  • ./: 현재 디렉토리
  • 폴더로 나눌 수 있으면 똑같은 파일명을 다른 폴더에 만들 수 있다.
  • 폴더는 부자관계가 있다.
  • <link rel="stylesheet" href="style.css"></link>를 만나면 파싱을 멈추고 css파일을 찾아 본 뒤, 없으면 다시 리퀘스트를 보낸다.
  • css는 요소에 스타일을 입혀야 되기 때문에 렌더링 이전에 있어야 한다. body 뒤에서 부르면 스타일 없이 그리고 한번 더 그린다.
  • 스크립트는 body 뒤에 있어야 요소를 선택할 수 있다.
  • 브라우저는 한줄 한줄 읽기 때문에 순서가 유의미하다.
  • ip: 인터넷에 매겨져 있는 컴퓨터의 번호
  • 서버는 그냥 소프트웨어다.
  • 내 로컬에 까는 서버가 localhost, path를 서버에 있는 것처럼 써도 된다.
  • http 1.0은 쪼개서 불러오고 http 2.0은 한방에 불러온다.
  • http는 단방향 통신이다. request의 기록이 안남는다.
  • 무전기는 한번 한번씩 얘기한다. 내 얘기가 끝나야 상대방이 얘기 가능.
  • 전화는 동시에 얘기 가능하다.
  • 기록이 안남으니까 요청이 또 올때 같은 요청이 와도 새롭다.
  • 그래서 세션, 토큰으로 클라이언트와 서버의 관계를 기록한다.
  • 단기적으로 기억할 내용은 메모리에 넣는다. (연산 결과를 cpu가 기억한다.)
  • getElementById를 하면 요소를 자료구조의 형태로 만들어서 메모리에 딱 넣는다.
  • Domtree, Cssomtree를 만든다는 것은 메모리 상에 넣는다는 의미
  • 구문은 무언가 일을 벌이는 문장
  • var : 변수를 생성하라는 명령
  • 키워드는 곧 명령이다.
  • 구문을 묶는 이유는 연결되서 수행되어져야하기 때문
